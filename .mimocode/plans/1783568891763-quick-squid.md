# 修复订阅转存串台问题

## 问题根因

`subscription_service.py:3825-3826` 中，当资源名是通用标签（如 "115资源 #2"、"S01E01-E11"）时，`_resource_has_identifiable_title()` 返回 False，系统直接 `return True` 放行。这意味着任何名称不含可识别影视标题的资源都会被无条件接受，导致串台。

**日志实证**：`[戴高乐之战：淬炼时代] 转存成功：115资源 #2` — 通用标签被直接放行。

## 修复方案

### 核心思路：转存前校验 115 分享内容

对于名称不含可识别标题的资源（pansou/TG 的通用标签），在转存前先获取 115 分享链接的文件列表，通过文件名校验确认内容与目标影视匹配。

### 修改文件

#### 1. `backend/app/services/subscription_service.py`

**A. 在 `_is_resource_relevant_for_subscription()` 中增加分享内容校验（第 3785 行）**

当 `_resource_has_identifiable_title(label)` 返回 False 时，不再直接 `return True`，改为尝试从资源的 `share_link` / `resource_url` 中获取 115 分享文件列表，用文件名做相关性校验：

```python
# 现有逻辑（第 3825-3826 行）：
if not self._resource_has_identifiable_title(label):
    return True  # ← 问题所在

# 改为：
if not self._resource_has_identifiable_title(label):
    # 尝试通过分享链接内容校验
    share_link = self._extract_share_link_from_item(item)
    if share_link:
        return await self._verify_share_content_matches_subscription(
            share_link, expected_title, expected_original_title, expected_year
        )
    return True  # 无分享链接时仍放行（兜底）
```

**B. 新增 `_extract_share_link_from_item()` 辅助方法**

从 item（dict 或 DownloadRecord）中提取 115 分享链接。

**C. 新增 `_verify_share_content_matches_subscription()` 方法**

1. 调用 `pan115_service.get_share_all_files_recursive(share_link)` 获取文件列表
2. 提取前 10 个视频文件名
3. 用 `tg_service._score_row_relevance()` 对每个文件名做相关性评分
4. 只要有一个文件名 `strong_hit && score >= 80`，认为匹配成功
5. 获取失败或无匹配 → 返回 False（拒绝转存）

**D. 将 `_is_resource_relevant_for_subscription` 改为 async**

因为需要调用 115 API 获取分享内容。

**E. 更新所有调用点**

- `_filter_resources_for_subscription()`（第 3830 行）→ 改为 async
- `_auto_save_resources()` 中的调用（第 2975 行）→ 加 await
- `_fetch_resources()` 中的调用（第 1725 行）→ 加 await

#### 2. `backend/app/services/subscription_service.py` — 增加 pansou 搜索时的预过滤

**在 `_fetch_from_pansou()` 返回结果后（第 1841 行和第 1887 行），增加基于资源名称的初步过滤**：

对 pansou 返回的每条资源，检查其 `title` 字段是否包含目标影视的关键词（中文标题/英文标题/年份），不含的直接剔除。这是一层轻量级的前置过滤，避免不相关的资源进入后续流程。

### 不修改的部分

- TG 搜索的 `score >= 80` 过滤已足够严格，不需改动
- HDHive 搜索结果自带 TMDB 关联，不需改动
- `_score_row_relevance()` 算法本身不需改动

## 验证方式

1. 部署后观察订阅扫描日志中的 `resource_relevance_filtered` 条目，确认通用标签资源被正确拦截
2. 确认合法资源（文件名包含目标影视标题）仍能正常转存
3. 检查 `_verify_share_content_matches_subscription` 的日志输出，确认分享内容校验逻辑工作正常
